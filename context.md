### 2. 제출물

- 데이터 모델 — ERD 또는 테이블 정의 (컬럼명, 타입, 제약조건, 인덱스)

기준: 다른 개발자(또는 AI)가 이 문서만 보고 추가 질문 없이 구현을 시작할 수 있는 수준

### 3. 시나리오

기술 스택: Java 또는 Kotlin + Spring Boot + PostgreSQL

#### S1. 선박 의약품 재고 관리 시스템

배경: 해양법(MLC 2006)에 따라 선박은 의무적으로 의약품을 비치해야 합니다.

필요한 기능:
1. 약품 카탈로그 관리
2. 선박별 의약품 비치 기준 관리
3. 재고 입고 (로트번호, 수량, 유효기한)
4. 약품 사용 기록 (FEFO: First-Expiry-First-Out)
5. 재고 현황 조회
6. 사용 이력 조회

만들지 않을 것: 발주/주문 관리, 비동기 처리/이벤트 버스/배치 작업, 인증/인가, 선종별 필수 약품 목록

고민해야 할 것들:
- 정책(최소 비치 기준)과 물리적 재고는 같은 개념일까요, 다른 개념일까요?
- 사용 기록은 어떤 입고 건에서 차감되었는지까지 추적해야 할까요?
- 재고가 부족한데 사용 요청이 들어오면?
- 유효기한이 지난 약품은 어떻게 처리할까요?

#### S2. 선원 익명 커뮤니티 게시판 (CHOSEN)

배경: 선원들이 해운사에 대한 경험을 공유하는 커뮤니티. 실명 채널과 익명 채널 두 종류.

필요한 기능:
1. 채널 관리 — 실명(REAL_NAME) 채널과 익명(ANONYMOUS) 채널
2. 게시글 CRUD — 페이지네이션 필요
3. 익명 닉네임 — 같은 사용자가 같은 게시글 내에서는 동일한 닉네임, 다른 게시글에서는 다른 닉네임
4. 댓글 & 답글 — 답글의 답글은 불가(최대 1단계 깊이). 삭제 시 답글이 있으면 "삭제된 댓글입니다"로 대체
5. 반응 (좋아요/싫어요) — 같은 반응 다시 누르면 취소, 다른 반응 누르면 변경. 중복 반응 불가
6. 조회수 — 게시글 상세 조회 시 증가

만들지 않을 것: 이미지 업로드, 알림 시스템, 관리자 블라인드/검수, 인증/인가 (X-User-Id 헤더), 실시간 (WebSocket/SSE), 검색

고민해야 할 것들:
- 익명 닉네임의 "범위"를 어떻게 모델링할까요?
- 두 사용자가 동시에 같은 게시글에 첫 댓글을 달면, 닉네임이 충돌할 수 있을까요?
- 좋아요/싫어요 토글은 어떻게 설계할까요?
- 삭제된 댓글에 답글이 달려있으면 어떻게 보여줘야 할까요?
- 게시글 목록에서 좋아요 수, 댓글 수를 매번 COUNT 쿼리로 구할까요?
